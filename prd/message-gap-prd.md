# MessageGap MVP PRD

日期：2026-06-15

## 1. 产品定位

MessageGap 是一个本地自用的 Next.js Agent 应用，用来持续发现互联网上刚出现的“信息差”信号，并支持基于这些信号另起 Agent 对话线程，继续打开来源页、搜索相似案例、追问风险和行动路径。

MVP 不是公开 SaaS，也不是多人协作系统。第一版只服务于你自己，重点是把“新信息发现 -> LLM 筛选 -> 本地沉淀 -> 引用信息继续追问”的闭环跑通。

## 2. MVP 目标

第一阶段由用户手动触发采集任务，搜索和爬取最近 3 日内的新内容，识别其中可能成为机会、红利、风险预警或行动窗口的信息差，并将有效结果保存到本地数据库。后续再接同 repo worker，实现每隔 4 小时自动运行一次。

核心目标：

- 新：优先发现刚发布几分钟到几小时内的帖子、公告、新闻、项目、规则变化。
- 准：不是普通热帖或泛泛提问，而是经过定制筛选 Skill 和 DeepSeek 分析后有潜在价值的早期信号。
- 可追溯：每条信息差必须保留来源链接、证据片段、发现时间和判断依据。
- 可追问：每条信息差都能开启独立 Agent 线程，继续基于来源页和搜索工具刨根问底。
- 有风险提示：允许收集“钻小空子”、灰色边缘、平台规则变化等内容，但必须明确提示法律、平台、资金和执行风险。

## 3. 目标用户与使用边界

目标用户：项目 owner 本人。

MVP 假设：

- 单用户、本地使用。
- 不需要账号系统。
- 不需要多端同步。
- 不需要个性化画像。
- 不需要通知推送。
- 不需要公开部署。
- 需要覆盖非常宽的信息源和机会类型。

## 4. 产品模块

### 4.1 信息采集模块

MVP 第一阶段先支持手动触发“立即采集”。后续 worker 接入后，再支持每 4 小时自动运行一次。

模块职责：

1. 按信息差分类生成搜索任务。
2. 通过浏览器搜索获取最近 3 日内的新内容，不接第三方搜索 API。
3. 对需要进一步确认的内容使用浏览器爬取或打开页面。
4. 抽取标题、正文、发布时间、作者、平台、链接、评论线索和证据片段。
5. 用 DeepSeek 和定制筛选 Skill 判断是否构成信息差。
6. 对候选内容进行分类、打分、风险识别和去重。
7. 将合格信息差保存到本地数据库。
8. 在前端看板中展示最新结果。

### 4.2 Agent 追问线程模块

第二个模块是 Agent 流对话。

用户可以从任意已保存的信息差创建一个新的调查线程。线程启动时自动引用：

- 信息差标题和摘要。
- 来源链接。
- 已抽取证据。
- 分类、标签、风险等级。
- 之前的采集和工具调用记录。

用户可以继续追问：

- “打开原帖，看看评论区有没有更多线索。”
- “这个信号为什么可能变成热点？”
- “过去 24 小时有没有类似帖子？”
- “这件事如果要行动，前 30 分钟应该做什么？”
- “有没有法律风险或平台封号风险？”
- “找几个历史相似案例对比。”

Agent 对话必须支持正常的工具调用生命周期：

- `tool_call_use`：Agent 开始调用工具时产生。
- `tool_call_end`：工具成功、失败或取消时产生。

前端需要展示这些事件，后端需要持久化这些事件，方便复盘 Agent 到底做了什么。

分类说明：

- 升学深造：保研、留学、交换、奖学金、夏令营、导师招生、学校项目、申请窗口。
- 大厂求职：实习、校招、社招、内推、HC 变化、岗位放量、笔试面试信息。
- 赚钱副业：现金流机会、副业模式、需求缺口、平台任务、低成本服务。
- 创业机会：新需求、新渠道、新品类、小生意、供需错配、商业模式验证信号。

- 技术开源：开源项目、开发者工具、AI/自动化红利、技术生态变化。
- 金融市场：市场异动、宏观政策、行业变化、风险提示，不输出个股买卖建议。
- 平台红利：平台规则变化、流量机制、内容分发、商家政策、冷启动机会。
- 合规风险：法规变化、监管动向、处罚案例、平台封禁规则、灰产风险预警。
- 城市资源：本地生活、城市服务、落户、消费券、场地、活动、社区资源。

## 6. 什么算“信息差”

信息差不是普通资讯，也不是普通热帖。它必须是一个新出现、可验证、可能带来机会或预警价值的早期信号。

### 6.1 硬性过滤规则

候选内容满足以下任意条件时，默认不保存：

- 没有可用发布时间，且无法推断新旧。
- 发布时间超过 3 天。
- 只是泛泛提问，没有事实、变化、需求、窗口期或异常信号。
- 只是观点输出，没有可执行含义。
- 没有来源链接或证据片段。
- 明确要求欺诈、盗号、黑客攻击、绕过监管、内幕交易、伪造材料等违法行为。

提问帖可以被保存，但必须满足一个前提：LLM 能从提问背后识别出可信的早期信号，例如重复出现的需求、新规则变化、平台异常、政策窗口、供需错配或市场异动。

### 6.2 信息差评分

每条候选内容生成 `opportunityScore`，范围 0-100。

MVP 推荐评分：

- 新鲜度 25 分：几分钟到几小时内最高，接近 3 天最低。
- 可行动性 20 分：是否能形成下一步动作。
- 信息不对称 15 分：是否隐藏在小圈层、冷门来源、评论区或新公告里。
- 窗口紧迫度 15 分：是否有截止时间、名额、先发优势或短期规则窗口。
- 证据强度 10 分：来源数量、来源可信度、证据是否具体。
- 潜在收益 10 分：对职业、升学、赚钱、创业、政策、风控的价值。
- 风险可识别度 5 分：风险是否能被清楚描述。

保存阈值：

- `>= 75`：高优先级信息差。
- `60-74`：候选信息差，证据足够时保存。
- `< 60`：默认丢弃。

纯提问帖默认最高不超过 60，除非有额外证据支撑。

## 7. 数据来源策略

MVP 支持两类采集方式：

1. Browser Search：通过浏览器打开搜索引擎，发现最近 3 日内的新 URL，不接第三方搜索 API。
2. 浏览器爬取：通过浏览器打开页面，抽取搜索结果无法覆盖的正文、评论或页面状态。

目标来源包括：

- Reddit。
- 小红书。
- 知乎。
- 微博。
- B 站。
- GitHub。
- 政府官网。
- 高校官网。
- 招聘网站。
- 企业招聘页。
- 新闻站点。
- 行业社区。
- 可访问的公众号文章。
- 其他搜索发现的公开页面。

采集约束：

- 只采集公开或用户自己可访问的内容。
- 不做高频、大规模、侵入式爬取。
- 对需要登录的平台，使用用户本机可访问的登录态浏览器会话进行爬取。
- 登录态只用于访问用户自己能正常看到的页面，不做绕过权限或绕过风控。
- 只保存证据、链接和必要摘要，不保存无关个人隐私。

## 8. DeepSeek 与定制 Skill 职责

DeepSeek 负责推理、筛选和整合，但不能作为唯一事实来源。事实必须来自搜索结果、浏览器页面、已保存证据或用户提供内容。

DeepSeek 任务：

- 生成不同分类下的搜索 Query。
- 扩展平台关键词和同义表达。
- 阅读抽取后的网页正文。
- 判断候选内容是否构成信息差。
- 分类和打标签。
- 生成摘要、价值判断和行动建议。
- 打信息差分数。
- 识别风险。
- 去重和合并相似信号。
- 在 Agent 线程里基于工具结果回答追问。

定制筛选 Skill 任务：

- 判断内容是不是“刚出现的信息差”，而不是普通新闻或普通提问。
- 判断信息是否可能形成热点、窗口期、套利空间、申请机会、求职机会、政策机会或风险预警。
- 判断证据是否足够。
- 判断是否存在误导、标题党、虚假机会或高风险执行路径。
- 给出保存、候选、丢弃三种结论。

建议将筛选规则做成可版本化配置，例如：

```text
src/skills/information-gap/
├── system.md
├── scoring.md
├── risk.md
├── categories.md
└── examples.json
```

## 9. 核心用户流程

### 9.1 定时采集流程

1. 用户手动启动一次 collection run；后续由 worker 每 4 小时自动启动。
2. 后端按 12 个分类生成搜索任务。
3. Browser Search 返回候选 URL。
4. 浏览器爬虫打开需要进一步确认的页面。
5. Extractor 抽取标题、正文、发布时间、作者、来源、链接和证据。
6. DeepSeek + 定制 Skill 评估候选内容。
7. DeduplicationService 合并相似信息。
8. 合格信息差写入数据库。
9. 看板按分数和新鲜度展示最新结果。

### 9.2 信息差查看流程

1. 用户打开首页看板。
2. 用户按分类、来源、风险等级、发布时间、分数筛选。
3. 用户点击一条信息差。
4. 详情页展示摘要、证据、来源链接、评分拆解、风险提示和行动建议。
5. 用户可以标记为有用、忽略、归档、需要追问。

### 9.3 Agent 追问流程

1. 用户在信息差详情页点击“深入追问”。
2. 系统创建或打开一个 Agent 调查线程。
3. 线程自动注入该信息差上下文。
4. 用户提出问题。
5. Agent 判断是否需要调用工具。
6. 前端展示 `tool_call_use`。
7. 后端执行 Browser Search、Browser Open、Page Extract、Local DB Lookup 等工具。
8. 工具结束后展示 `tool_call_end`。
9. Agent 基于工具结果和已有证据回答。
10. 线程消息、引用和工具事件全部保存到本地。

## 10. 功能需求

### 10.1 首页看板

首页需要展示：

- 最新高优先级信息差。
- 分类筛选。
- 来源筛选。
- 新鲜度筛选。
- 风险等级筛选。
- 搜索框。
- 当前采集状态。
- 上次成功采集时间。
- 最近一次失败原因。
- 手动“立即采集”入口。

信息差卡片字段：

- 标题。
- 分类。
- 来源。
- 发布时间。
- 发现时间。
- 信息差分数。
- 风险等级。
- 一句话摘要。
- 证据数量。
- 行动窗口。

### 10.2 信息差详情页

详情页需要展示：

- 完整摘要。
- 为什么值得关注。
- 证据列表。
- 来源 URL。
- 原始发布时间。
- 分类和标签。
- 评分拆解。
- 机会窗口。
- 建议行动。
- 风险提示。
- 相似信息差。
- “深入追问”按钮。

### 10.3 采集任务记录

每次采集任务需要保存：

- Run ID。
- 开始时间。
- 结束时间。
- 状态。
- 搜索过的来源。
- 发现的 URL 数量。
- 打开的页面数量。
- 评估的候选数量。
- 保存的信息差数量。
- 错误列表。
- DeepSeek token 和成本估算，如果 API 可提供。

### 10.4 Agent 对话

Agent 对话需要支持：

- 每条信息差可以有独立线程。
- 自由追问。
- 基于来源证据回答。
- 展示工具调用过程。
- 保存 `tool_call_use` 和 `tool_call_end`。
- 回答中引用来源 URL 和证据片段。
- 线程历史本地保存。
- 从信息差详情页重新打开旧线程。

### 10.5 风险处理

系统允许记录灰色边缘、钻小空子、平台规则变化、补贴窗口、流量红利等内容，但必须对每条高风险内容展示：

- 风险等级：low、medium、high、critical。
- 法律和合规风险。
- 平台封禁或账号风险。
- 资金损失风险。
- 声誉风险。
- 证据不确定性。
- 是否不建议执行。

系统不能生成欺诈、黑客攻击、盗号、绕过平台风控、内幕交易、伪造材料等违法或明显有害操作步骤。

### 10.6 金融市场内容

金融市场分类只做信息监测和风险分析，不提供投资建议。

允许输出：

- 发生了什么。
- 为什么可能重要。
- 证据是什么。
- 风险是什么。
- 后续应该观察什么。

不输出：

- 明确买入、卖出、加杠杆建议。
- 保证收益。
- 内幕交易或规避监管方案。

## 11. Agent 工具调用事件规范

工具调用事件必须能支持流式 UI。

### 11.1 `tool_call_use`

当 Agent 准备使用工具时产生。

示例：

```json
{
  "eventType": "tool_call_use",
  "toolName": "web_search",
  "status": "running",
  "inputPreview": {
    "query": "site:reddit.com subsidy opportunity last 24 hours"
  },
  "startedAt": "2026-06-15T02:33:00.000Z"
}
```

### 11.2 `tool_call_end`

当工具成功、失败或取消时产生。

示例：

```json
{
  "eventType": "tool_call_end",
  "toolName": "web_search",
  "status": "success",
  "outputPreview": {
    "resultCount": 5,
    "topUrls": ["https://example.com/post"]
  },
  "endedAt": "2026-06-15T02:33:08.000Z"
}
```

状态枚举：

- `running`
- `success`
- `error`
- `cancelled`

前端展示原则：

- 工具开始时显示正在调用哪个工具。
- 工具结束后显示简要结果。
- 错误要能展开查看。
- 默认隐藏敏感输入，例如 API Key、Cookie、完整登录态信息。

## 13. 推荐技术架构

MVP 技术选型：

- Next.js App Router：前端页面、后端 Route Handlers、服务端逻辑都放在同一个 Next 项目中。
- React：前端 UI 统一使用 React 组件。
- TypeScript：前端、后端、脚本、数据模型全部使用 TS。
- shadcn/ui：作为基础 UI 组件来源。
- Tailwind CSS：作为样式系统。
- Node.js Worker：在同一个 Next repo 内执行定时采集、爬取、LLM 评估，不引入独立 Express/Nest 后端。
- SQLite：本地自用数据库。
- Drizzle ORM：数据库 schema、迁移和类型安全查询。
- Playwright：浏览器搜索、登录态爬取和页面抽取。
- Browser Search：通过 Playwright 驱动浏览器搜索，不接搜索 API。
- DeepSeek Client：封装 LLM 请求。
- Local File Storage：可选保存原始抽取结果。

推荐项目结构：

```text
messageGap
├── app
│   ├── page.tsx
│   ├── signals
│   ├── threads
│   └── api
├── src
│   ├── agent
│   ├── components
│   ├── crawler
│   ├── db
│   ├── deepseek
│   ├── features
│   ├── lib
│   ├── scheduler
│   ├── scoring
│   ├── search
│   ├── skills
│   └── sources
├── drizzle
├── prd
└── scripts
```

后端核心服务：

- SchedulerService：第一阶段只暴露手动触发；后续 worker 接入后每 4 小时启动任务。
- SearchOrchestrator：生成并执行搜索任务。
- BrowserCrawler：打开和抽取页面。
- SourceExtractor：规范化页面内容。
- SignalEvaluator：调用 DeepSeek 和筛选 Skill。
- DeduplicationService：合并相似候选。
- RiskAnalyzer：生成风险等级和提示。
- ThreadAgent：运行追问对话。
- ToolRuntime：产生 `tool_call_use` 和 `tool_call_end`。

代码组织约束：

- 不写几千行单文件，不把页面、状态、请求、业务逻辑和 UI 全塞进一个文件。
- 页面文件只负责路由级组合和数据入口，复杂 UI 拆到 feature components。
- API Route 只负责请求解析、校验和调用 service，不直接写大段业务流程。
- 采集、评分、爬取、Agent、数据库访问分别拆成独立模块。
- shadcn/ui 组件放在 `src/components/ui`，业务组件放在 `src/features/*/components`。
- 类型定义靠近业务模块，跨模块共享类型再放到 `src/types` 或 `src/lib`。
- 单个业务组件建议控制在 200 行以内；超过时优先拆子组件、hook、service 或纯函数。

## 14. MVP 范围

### 14.1 In Scope

- 本地单用户 Next.js 应用。
- 手动立即采集。
- 预留后续 4 小时 worker 定时采集。
- 12 个一级分类。
- 仅保存 3 日内新内容。
- Browser Search 抽象层。
- 浏览器爬取抽象层。
- DeepSeek 接入。
- 定制信息差筛选 Skill。
- 信息差评分和过滤。
- 看板页面。
- 详情页。
- Agent 追问线程。
- `tool_call_use` / `tool_call_end` 展示和持久化。
- Agent 响应流使用 SSE。
- SQLite 本地持久化。
- 风险提示系统。

### 14.2 Out Of Scope

- 用户账号。
- 团队协作。
- 云同步。
- 付费系统。
- 移动端 App。
- 推送通知。
- 个性化画像推荐。
- 自动执行机会。
- 直接投资建议。
- 违法或明显有害操作指导。
- 大规模爬虫基础设施。

## 15. 非功能需求

- 本地优先：数据主要保存在本机。
- 可追溯：每条结论必须能追到来源和证据。
- 可恢复：单个来源失败不能导致整次任务失败。
- 可配置：来源、分类、搜索频率、评分阈值、Skill 规则可调整。
- 可观察：采集 run、错误、工具调用、LLM 输出都能查看。
- 克制爬取：避免高频请求和对目标平台造成压力。

## 16. 验收标准

MVP 达标条件：

- 应用可以在本地启动。
- 用户可以手动触发采集。
- 第一阶段可以手动触发采集。
- 系统只保存最近 3 日内的信息差。
- 每条信息差有分类、来源 URL、证据、评分、摘要和风险等级。
- 看板按分数和新鲜度展示结果。
- 用户可以打开详情页。
- 用户可以从详情页创建 Agent 追问线程。
- Agent 回答基于信息差上下文和来源证据。
- 工具调用过程展示 `tool_call_use` 和 `tool_call_end`。
- 灰色边缘或高风险机会有明确风险提示。
